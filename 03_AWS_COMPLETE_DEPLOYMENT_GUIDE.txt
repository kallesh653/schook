1.3 AWS GLOBAL INFRASTRUCTURE
--------------------------------------------------------------------------------

HIERARCHY:

1. REGIONS (31 worldwide)
   Geographic areas with multiple data centers

   Example Regions:
   - us-east-1 (N. Virginia, USA)
   - us-west-2 (Oregon, USA)
   - eu-west-1 (Ireland)
   - ap-south-1 (Mumbai, India)
   - ap-southeast-1 (Singapore)

   Each region is isolated and independent

2. AVAILABILITY ZONES (99 total)
   Multiple isolated data centers within a region

   Example (us-east-1 has 6 AZs):
   - us-east-1a
   - us-east-1b
   - us-east-1c
   - us-east-1d
   - us-east-1e
   - us-east-1f

   WHY Multiple AZs?
   If one data center fails ‚Üí Other AZs still work
   HIGH AVAILABILITY! ‚úì

3. EDGE LOCATIONS (400+ worldwide)
   Cache content close to users
   Used by CloudFront (CDN)

IMPORTANT: ROOT USER vs IAM USER

ROOT USER (What you created):
- Full access to everything
- Can't be restricted
- Should ONLY be used for:
  - Initial setup
  - Billing/account management
  - Creating IAM users

IAM USER (What you should use daily):
- Created by root user
- Can have restricted permissions
- Safer for daily use

BEST PRACTICE:
1. Login as root user
2. Create IAM user with admin permissions
3. Logout root user
4. Use IAM user for everything
5. Enable MFA on both accounts

2.1 EC2 (ELASTIC COMPUTE CLOUD)
--------------------------------------------------------------------------------

WHAT IS EC2?

EC2 = Virtual servers in the cloud
Like having a computer (Windows/Linux) that you can:
- Start in seconds
- Access remotely
- Scale up/down
- Pay by the hour

SIMPLE ANALOGY:
Your laptop = Physical computer
EC2 instance = Virtual computer in Amazon's data center

USE CASES:
‚úì Web servers (host your backend)
‚úì Application servers
‚úì Development/test environments
‚úì Big data processing
‚úì Gaming servers


KEY CONCEPTS:

1. INSTANCE
   Running virtual machine
   Example: t3.micro, t3.medium

2. AMI (Amazon Machine Image)
   Template for instance
   Like Windows installation disk
   Contains: OS + pre-installed software

3. INSTANCE TYPE
   Hardware configuration
   Example: t3.medium = 2 vCPUs, 4 GB RAM

4. EBS (Elastic Block Store)
   Virtual hard drive
   Attached to instance

5. SECURITY GROUP
   Virtual firewall
   Controls inbound/outbound traffic


INSTANCE TYPES (Categories):

1. GENERAL PURPOSE (T, M series)
   Balanced CPU, memory, network
   Use: Web servers, small databases
   Examples: t3.micro, t3.small, m5.large

2. COMPUTE OPTIMIZED (C series)
   High CPU performance
   Use: Video encoding, batch processing
   Examples: c5.large, c5.xlarge

3. MEMORY OPTIMIZED (R, X series)
   Large memory
   Use: In-memory databases, big data
   Examples: r5.large, x1e.xlarge

4. STORAGE OPTIMIZED (I, D series)
   High disk throughput
   Use: Data warehousing, Hadoop
   Examples: i3.large, d2.xlarge

5. GPU INSTANCES (P, G series)
   Graphics processing
   Use: Machine learning, 3D rendering
   Examples: p3.2xlarge, g4dn.xlarge


PRICING MODELS:

1. ON-DEMAND
   Pay by the hour/second
   No commitment
   Most expensive
   Use: Short-term, unpredictable workloads

   Example: t3.medium = $0.0416/hour = $30/month

2. RESERVED INSTANCES
   1 or 3 year commitment
   Save 40-75%
   Pay upfront or monthly
   Use: Steady-state workloads

   Example: t3.medium Reserved = $0.0208/hour = $15/month
   Savings: 50%!

3. SPOT INSTANCES
   Bid on spare capacity
   Save up to 90%
   Can be terminated anytime
   Use: Flexible, fault-tolerant workloads

   Example: t3.medium Spot = $0.0125/hour = $9/month
   Savings: 70%!

4. SAVINGS PLANS
   Flexible commitment (1 or 3 years)
   Save up to 72%
   Use: Consistent usage, any instance type

2.4 VPC (VIRTUAL PRIVATE CLOUD)
--------------------------------------------------------------------------------

WHAT IS VPC?

VPC = Your own private network in AWS
Isolated network environment
Complete control over:
- IP address range
- Subnets
- Route tables
- Network gateways

SIMPLE ANALOGY:
VPC = Your private office building
- Subnets = Different floors
- Internet Gateway = Main entrance
- Security Groups = Security guards
- Route Tables = Directions/signs


DEFAULT VPC:

Every AWS account has default VPC (per region)
- CIDR: 172.31.0.0/16
- Subnets in each AZ
- Internet Gateway attached
- Default Security Group

You can use default VPC or create custom VPC


KEY COMPONENTS:

1. CIDR BLOCK
   IP address range for VPC
   Example: 10.0.0.0/16 = 65,536 IP addresses

2. SUBNET
   Subdivision of VPC
   Public subnet: Has internet access
   Private subnet: No direct internet access

3. INTERNET GATEWAY
   Allows VPC to access internet
   Attached to VPC

4. NAT GATEWAY
   Allows private subnet to access internet
   But internet can't initiate connection to private subnet

5. ROUTE TABLE
   Rules for routing traffic
   Where to send packets

6. SECURITY GROUP
   Virtual firewall for instances
   Stateful (return traffic allowed automatically)

7. NACL (Network ACL)
   Firewall for subnets
   Stateless (must explicitly allow return traffic)


TYPICAL VPC ARCHITECTURE:

VPC: 10.0.0.0/16

‚îÇ   ‚îú‚îÄ‚îÄ Internet Gateway attached
‚îÇ   ‚îú‚îÄ‚îÄ NAT Gateway
‚îÇ   ‚îî‚îÄ‚îÄ Load Balancer
‚îú‚îÄ‚îÄ Private Subnet A (10.0.2.0/24)
‚îÇ   ‚îú‚îÄ‚îÄ EC2 Instances (Web servers)
‚îÇ   ‚îî‚îÄ‚îÄ Can access internet via NAT Gateway
‚îî‚îÄ‚îÄ Private Subnet B (10.0.3.0/24)
    ‚îú‚îÄ‚îÄ RDS Database
    ‚îî‚îÄ‚îÄ No internet access


WHY PUBLIC AND PRIVATE SUBNETS?

PUBLIC SUBNET:
- Resources need internet access
- Can be accessed from internet
- Examples: Load Balancers, NAT Gateways, Bastion hosts

PRIVATE SUBNET:
- Resources don't need direct internet access
- Can't be accessed from internet (secure!)
- Can access internet via NAT Gateway
- Examples: Application servers, Databases

SECURITY BEST PRACTICE:
‚úì Web servers ‚Üí Private subnet (behind Load Balancer)
‚úì Databases ‚Üí Private subnet (no internet access)
‚úì Only Load Balancer ‚Üí Public subnet
